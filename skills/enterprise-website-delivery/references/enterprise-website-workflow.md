# 企业官网静态 HTML 生成工作流

## 1. 工作流目标

当用户提供一段公司简介后，AI 执行者根据企业官网建设流程自动完成：

- 判断是否缺少公司名称、联系方式、主营业务、地址、目标客户等必要信息。
- 对企业官网必要信息必须先追问，不得直接用占位文案跳过。
- 根据企业类型自动选择视觉与结构参考模板。
- 在生成前明确输出所选模板、页面栏目、图片清单和缺失信息处理结果。
- 自动规划页面栏目与网站内容。
- 自动生成匹配行业的官网图片。
- 输出纯静态 HTML 网站，不使用后端、不使用数据库。
- 检查页面、跳转、图片、样式、模板符合度是否完整。

最终交付一个可以直接打开的静态网站目录。

## 1.1 强制执行原则

这份工作流是生成企业官网时的最高执行依据。AI 执行者不得因为上下文里存在旧网站、旧样式、旧文件或旧生成习惯，就复用之前的页面结构。

### 最高优先级：先像一个好看的真实官网

本工作流的核心目标是交付一个“打开后像真实企业官网、审美舒服、信息清楚、能获客”的静态网站。流程回执、验收报告、图片清单只是防止遗漏的证据，不能替代网站本身的视觉质量。

如果出现以下情况，即使 `_workflow/`、`_verification/`、`image-manifest.json` 都存在，也必须判定失败并重做：

```text
1. 首屏像通用插画落地页，不像企业官网。
2. 全站主要图片都是扁平插画、抽象 UI 图、图表卡通图。
3. 页面看起来像“为了满足检查清单而生成”，不是自然的官网设计。
4. 文字很大但画面没有企业质感、图片没有业务现场感。
5. 证据文件声称“半写实/真实场景”，但实际图片明显是扁平插画。
6. 页面不如没有使用本工作流时生成的版本自然、可信、舒服。
```

执行顺序必须是：

```text
1. 先做官网视觉和内容。
2. 再做证据文件和验收。
3. 证据文件不得反向污染页面设计。
```

### 页面版式硬规则：企业官网必须横向撑满

企业官网页面必须有横向铺开的真实官网气质，不能像内容被塞在中间窄容器里。

```text
1. 页面主视觉、背景色块、分区背景、轮播图、底部区域必须使用 full-width / full-bleed 版式，横向撑满浏览器宽度。
2. 文字内容可以使用 1200px-1440px 的内层容器保证阅读舒适，但外层 section 背景、图片和色块必须延展到整屏。
3. 首屏优先使用整屏宽幅大图、背景图、左右分栏大图或轮播图，不能只是中间一块小卡片。
4. 服务型企业也可以使用大背景图 + 左侧文字遮罩 / 渐变遮罩 / 底部服务入口条，让页面更像真实官网。
5. 制造和工厂模板必须有大图、宽幅车间/设备/产线/产品展示区，不能全靠文字卡片表达实力。
6. 禁止全站使用同一个居中窄容器导致左右留白过大；如果出现 1440px 以上桌面端大面积空白，视为版式失败。
7. 禁止连续多个纯白 section + 小卡片；至少穿插品牌色浅底、深色区、宽幅图片区、图文交错区或轮播区。
```

每次生成前必须完成六件事：

```text
1. 读取并遵守本工作流。
2. 检查当前 AI 执行环境是否具备企业官网交付所需能力，缺失则说明替代方案或停止。
3. 判断企业类型并锁定一个模板。
4. 检查必要信息是否缺失，缺失则先追问。
5. 输出“设计执行卡”：设计依据、首屏构图、视觉系统、图片策略、动效策略、禁止事项。
6. 输出“生成前确认摘要”：模板、页面、图片数量、缺失信息处理方式。
```

如果没有完成以上六件事，不得开始生成 HTML、CSS、JS 或图片。

## 1.2 读取与执行机制

这份工作流不得只作为“背景参考”泛读。AI 执行者每次执行企业官网生成任务时，必须按阶段读取、复述、执行和验收。

### 读取编码规则

本文件是 UTF-8 编码。若在 Windows PowerShell 中读取，必须使用：

```powershell
Get-Content -Encoding UTF8 -LiteralPath "references/enterprise-website-workflow.md"
```

禁止使用未指定 `-Encoding UTF8` 的普通 `Get-Content` 来理解本工作流；如果读取后出现乱码，视为工作流未读取成功，不得继续生成官网。

### 阶段 0：工作流读取确认

开始任何生成前，必须先输出：

```text
【工作流读取确认】
已读取工作流文件：
本次任务将执行的关键章节：
- 信息追问规则
- 企业类型判断
- 固定参考模板库
- 图片生成规则
- 静态网站输出结构
- 生成前确认策略
- 验收标准
```

如果没有读取工作流文件，或者无法读取工作流文件，必须明确说明，不能假装已经读取。

### 阶段 0.5：执行环境与能力检查

本工作流是通用流程，不绑定 Codex、Antigravity、Cursor、Claude Code、Gemini CLI 或其他任何单一工具。AI 执行者必须先检查当前环境能否完成任务，而不是假装具备能力。

必需能力：

```text
1. 文件读写能力：能创建 HTML、CSS、JS、图片目录和交付报告。
2. 前端实现能力：能编写静态多页面官网，并保证移动端适配。
3. 视觉设计能力：能根据三套模板方向形成设计执行卡，避免白底卡片堆叠。
4. 图片处理/生成能力：能使用用户真实图片，或生成/准备符合行业的官网图片。
5. 浏览器预览能力：能打开本地 HTML 或启动静态服务进行截图检查。
6. 验收能力：能检查链接、图片、响应式、AIO 文件和模板符合度。
```

输出格式：

```text
【执行环境检查】
当前工具/环境：
文件读写：可用 / 不可用
前端实现：可用 / 不可用
视觉设计参考：可用 / 不可用 / 使用内置审美规则
图片生成/处理：可用 / 不可用 / 需要用户提供图片
浏览器预览/截图：可用 / 不可用
处理方式：
```

执行规则：

- 如果没有文件读写能力，不能继续生成官网。
- 如果没有浏览器预览/截图能力，必须明确说明，并把“截图验收”标记为未完成，不能声称视觉已验收。
- 如果没有图片生成能力，也没有用户真实图片，必须说明图片素材限制；不得用纯色块、占位块或扁平图表冒充真实官网图片。
- 如果当前工具有内置设计 skill、审美规则、设计系统或前端 UI 能力，必须使用；如果没有，就按本工作流中的视觉底线和三套模板方向执行。
- 如果当前环境是 Codex 且已安装本地 skill，可以额外使用 `enterprise-website-delivery`、`design-taste-frontend`、`ui-ux-pro-max`；这些是 Codex 环境增强项，不是本工作流的通用前提。
### 阶段 1：参考源证据检查

凡是工作流或用户输入中出现的参考模板链接、参考网站、图片、文件、Logo、公司资料，AI 执行者必须先做参考源检查。

输出格式：

```text
【参考源检查】
工作流文件：已读取 / 未读取 / 无法读取
用户资料文件：已读取 / 未提供 / 无法读取
参考模板链接：已打开 / 未打开 / 无法访问
用户提供图片：已查看 / 未提供 / 无法查看
```

硬规则：

- 没有打开过的链接，不得说“我看了这个模板”。
- 没有查看过的图片，不得判断图片内容。
- 没有读取过的文件，不得声称已按文件内容执行。
- 如果参考链接无法访问，只能根据工作流里的文字描述执行，并必须在生成前确认摘要中说明“参考链接未实地查看”。
- 如果用户要求“参考某个链接的效果”，AI 执行者必须先打开链接或截图检查；打不开时必须暂停说明，不能凭想象复刻。

### 阶段 2：本次强制执行清单

生成前必须把本次任务压缩成一份不超过 15 条的执行清单，并在生成完成后逐项验收。

推荐格式：

```text
【本次强制执行清单】
1. 企业必要信息是否完整：
2. 是否需要追问：
3. 企业类型：
4. 所选模板：
5. 页面文件：
6. 首页板块：
7. 图片数量与文件名：
8. 是否优先使用真实图片：
9. 是否禁止虚构新闻/客户/证书：
10. SEO/AIO 文件：
11. 桌面端截图检查：
12. 移动端截图检查：
```

### 阶段 2.5：设计执行卡

在选择模板后、生成文件前，必须输出设计执行卡。设计执行卡是防止页面太素、太像白底模板页的关键约束。

输出格式：

```text
【设计执行卡】
已使用/检查的设计依据：
Design Read：
企业官网模板：
页面栏目：
首屏构图：
视觉关键词：
配色系统：
字体系统：
图片策略：
动效策略：
禁止事项：
移动端策略：
```

硬规则：

- 设计依据只能优化视觉表达、设计系统和交互体验，不得改变已选企业官网模板的页面栏目、图片清单、输出结构和行业定位。
- 不允许只输出“商务、大气、简洁”这类空泛词，必须明确首屏构图、色彩、字体、图片和动效。
- 如果没有输出设计执行卡，不得生成 HTML、CSS、JS 或图片。
- 设计执行卡必须服务于最终页面美观，不得变成机械 checklist。网站页面中禁止出现“工作流、验收、证据、清单”等交付过程内容。

只有完成“工作流读取确认”“执行环境检查”“参考源检查”“本次强制执行清单”“设计执行卡”和“必要信息追问”后，才允许进入生成 HTML、CSS、JS 和图片的阶段。

如果缺少任一阶段输出，视为工作流执行失败，必须停止并补齐。

### 阶段 2.6：受控配色与主题色落位

企业官网不得每次默认蓝色，也不得为了避免蓝色而随机配色。AI 执行者必须从品牌来源或本工作流的固定色板中选择一套，并把主题色真实融入页面组件。

配色优先级：

```text
1. 用户提供的 logo、VI、品牌色。
2. 用户提供的参考网站主色。
3. 本工作流的行业色板库。
4. 若仍无法判断，选择该行业最稳妥但不重复的色板，并说明原因。
```

设计执行卡中的“配色系统”必须写成：

```text
色板名称：
选择原因：
主色：
辅助色：
强调色：
背景色：
文字色：
禁用颜色：
主题色落位：
```

禁止只写“蓝色、白色、大气”这类空泛描述。不得临时发明一套没有名称、没有使用位置、没有选择理由的颜色。

#### 主题色落位规则

主题色不得只停留在 CSS 变量中，也不得只用于一个按钮。每个网站至少在以下位置使用主题强调色：

```text
1. 当前导航下划线、选中态或顶部细线。
2. 首页 H1 中 1-2 个关键词高亮。
3. 主 CTA 按钮背景或高对比描边。
4. 次级按钮 icon、箭头、边框 hover 或文字 hover。
5. section eyebrow、小标题、分区标题装饰线。
6. 服务/产品/优势卡片的 icon、序号、角标或 hover 状态。
7. 图片遮罩、轮播点、进度条、局部光效或信息条。
8. 页脚重点链接、联系按钮或二维码说明点缀。
```

主题主色必须进入以下位置：

```text
1. 顶部导航、首屏深色区域、页脚或重要分区背景中的至少两处。
2. 首屏图片遮罩或渐变层。
3. 标题文字、深色背景、宽幅统计条或栏目背景。
```

使用比例建议：

```text
主色约 60%，中性色/背景色约 30%，强调色约 10%。
```

禁止：

```text
1. 只把主题色用在一个按钮上。
2. 所有元素都刷成强调色。
3. 每个 section 换一套颜色。
4. 大面积彩虹色、多重高饱和渐变或随机撞色。
5. 未经用户要求整站大面积紫蓝渐变。
6. 用户没有品牌色时，所有行业都默认科技蓝。
```

## 1.3 禁止行为

执行本工作流时，禁止以下行为：

- 未读取工作流文件，却声称已按工作流执行。
- 未检查执行环境能力，却直接开始生成官网。
- 明明缺少图片生成、浏览器预览或文件写入能力，却没有说明限制或替代方案。
- 未输出设计执行卡，却开始生成 HTML、CSS、JS 或图片。
- 未打开参考模板链接，却声称已参考链接视觉效果。

- 图片数量没有达到模板最低要求，却声称图片完整。

## 3. 信息追问规则

AI 执行者收到公司简介后，先判断企业官网必要信息是否完整。

### 必须优先确认的信息

- 公司名称
- 主营业务或核心服务
- 企业所属类型：服务行业、制造行业、企业工厂/工业企业
- 至少一种有效联系方式：电话、微信、邮箱或表单入口
- 所在城市或服务区域
- 详细地址：如果是到店服务、工厂参观、线下交付类企业，则必须补充；如果只是线上获客，可先使用城市/服务区域

如果缺少这些信息，必须追问。

### 联系方式追问规则

联系方式属于企业官网必要信息，但不强制同时具备电话、邮箱和详细地址。

AI 执行者必须确认至少一种真实有效的触达方式，并确认城市或服务区域。电话或微信优先；如果用户提供邮箱、表单入口或其他联系方式，也可以作为有效联系方式。

如果完全缺少联系方式，AI 执行者必须先追问，不允许默认使用占位符。

推荐追问：

```text
官网需要展示至少一种真实联系方式。请补充联系电话、微信、邮箱或其他有效联系方式，并补充所在城市/服务区域；如果暂时没有，请明确回复“允许联系方式占位”，我再继续生成。
```

只有当用户明确说“允许占位”“先占位”“没有就先空着”等意思时，才可以使用：

```text
请补充企业联系方式
请补充所在城市或服务区域
```

### 可以在用户允许后占位的信息

- 微信
- 地图
- logo
- 真实案例
- 新闻资讯
- 资质荣誉

### 追问方式

一次只问必要问题，不要一次性抛出大量问题。

示例：

## 5. 固定参考模板库

注意：模板只作为视觉风格和页面结构参考，不直接复制源码、图片、文案或受版权保护的素材。最终网站由 AI 执行者重新生成静态 HTML、CSS、JS 和图片。

### 模板执行硬规则

选定模板后，AI 执行者必须执行对应模板的页面栏目、首页板块、图片数量和视觉特征，不得退回到通用企业站结构。

如果模板包含参考链接，AI 执行者必须先打开或检查参考链接，再描述该模板的实际视觉特征。若无法打开，只能使用本工作流中写明的文字规则，不得编造“已查看链接后的视觉结论”。

模板不是“只借一个颜色和栏目名称”。必须从参考模板提炼布局骨架和视觉节奏，再用本公司内容重新生成。至少要明确并执行：

```text
1. 首屏是大图、左右分栏、轮播、卡片叠层还是大横幅。
2. 导航是否是多页面企业站导航，而不是单页落地页导航。
3. 图片在页面中的密度和尺寸：首屏大图、栏目配图、图文交错、列表缩略图等。
4. 中段是否有图文交错、产品/服务列表、流程、案例/应用、资讯，而不是连续白底卡片。
5. 底部是否有企业信息、联系方式、快速入口和备案/版权位。
6. 页面是否横向撑满：背景、首屏、图片区、色块、底部必须铺满浏览器宽度。
7. 是否有真实官网常见的宽幅模块：轮播 banner、整屏背景图、横向服务入口条、大图文交错区等。
```

如果最终结果只是一屏文字加若干白底卡片，或只使用同一类抽象插画填充所有图片位，视为没有执行模板。


如果最终页面栏目、图片数量或视觉方向和所选模板不一致，视为生成失败，必须重做。

## 6. 默认页面栏目

AI 执行者应根据行业自动调整页面名称，但必须保留完整企业官网结构。

### 默认五大类导航规则

除非用户明确要求更多一级栏目，企业官网公开导航默认只保留五大类，避免把服务、方案、场景、案例拆得过散：

```text
1. 首页
2. 公司介绍
3. 业务与方案（可按行业命名为服务与方案、产品与能力、服务/产品与方案等）
4. 企业新闻
5. 联系我们
```

执行要求：

- 第 3 类必须合并承载服务项目、产品/能力、解决方案、应用场景、案例/项目等业务内容；缺少真实案例时，用“应用场景”表达，不单独生成案例一级栏目。

## 8. 图片生成规则

AI 执行者自动生成官网图片，图片必须与企业行业匹配。

图片要求：

```text
无文字
无 logo
无水印
适合网页裁切
风格统一
文件保存到 assets/images/
```

默认生成图片数量：

```text
服务行业：至少 8 张
制造行业：至少 10 张
企业工厂/工业企业：至少 10 张
```

图片文件名必须使用所选模板对应的图片清单，不得只生成 hero/about/service/case 四张通用图片。

真实图片模式：如果用户提供真实图片，必须优先使用真实图片；AI 图片只用于补充缺失素材，并且生成数量仍需达到所选模板最低要求。

AI 生成图片必须符合公司简介和行业场景，数量不能过少，风格必须统一；不得生成带文字、logo、水印、明显虚假证书、虚假客户 logo 或明显不可信的企业实拍内容。

图片质量硬规则：

```text
1. 图片不是“有文件就算完成”，必须服务于页面内容和模板结构。
2. 首屏图片必须能一眼表达企业业务，不得使用过于抽象、看不出行业的通用插画。
3. 服务行业可以使用办公、沟通、交付、数据看板、团队协作、客户咨询等真实感或半写实图片；抽象 UI 插画最多作为辅助图，不得全站都是抽象图。
4. 制造行业必须包含产品、设备、加工、检测、应用场景或技术细节等工业语义。
5. 企业工厂必须包含厂区、车间、产线、设备、仓储、交付、质检等工厂语义。
6. 不得生成假客户 logo、假证书、假奖项、假新闻现场。
7. 如果无法调用图片生成工具，必须明确说明“无法生成图片”，不能用占位块或纯 CSS 图形冒充已生成图片。
8. `image-manifest.json` 中写“真实感/半写实”，实际图片却是扁平插画或抽象图，视为图片验收失败。
9. 首屏主图禁止使用只有图表、窗口、圆点、人物剪影的扁平插画；服务型企业首屏应优先使用办公、服务交付、客户沟通、真实业务场景。
10. 禁止多处重复使用同一张图片或同一构图图片；同一主体、同一会议室、同一人物组合不得反复出现。
11. 禁止全站图片都是真人会议/办公图；必须混入空间、环境、细节、小配图或行业场景，让页面节奏不疲劳。
```

每张图片必须进入 `assets/images/image-manifest.json`，字段至少包含：

```json
{
  "file": "hero.png",
  "page": "index.html",
  "slot": "首页首屏",
  "source": "user-provided | ai-generated",
  "style": "真实感 / 半写实 / 产品图 / 场景图 / 图形辅助",
  "prompt": "生成或处理图片时使用的中文提示词",
  "alt": "图片 alt 文案",
  "why": "为什么这张图适合该页面位置"
}
```

如果用户没有提供真实图片，图片策略应优先生成“行业可信的场景图/产品图/空间图”，而不是只生成抽象图形。

### 图片类型配比规则

生成前必须先做图片分镜表，而不是一次性生成一批相似图片。不同模板的图片应按下列类型混合：

服务行业至少包含：

```text
1. 首屏宽幅业务场景图 1 张：真实感办公、服务交付、客户沟通或数据运营场景。
2. 服务/解决方案场景图 2-3 张：可以有人，但不能全是同一类会议照片。
3. 空间/环境/城市/办公景色图 1-2 张：用于分区背景、关于我们、联系页面或过渡区。
4. 细节小配图 1-2 张：电脑屏幕、白板、资料、手部操作、工作台、植物、建筑局部等。
5. 图形辅助图最多 1-2 张：只能作为小配图或背景纹理，不得做首屏主视觉。
```

制造行业至少包含：

```text
1. 首屏宽幅设备/产品/工艺场景图。
2. 产品或设备特写图。
3. 生产/加工/装配场景图。
4. 质检/检测/参数/材料细节图。
5. 应用场景或交付包装图。
```

企业工厂/工业企业至少包含：

```text
1. 首屏宽幅厂区/车间/产线/设备图。
2. 车间或产线大图。
3. 设备、工艺、仓储、物流、质检等细节图。
4. 工厂外观、园区、道路、装卸、包装或交付图。
5. 证书墙只能用抽象“资质展示区”或占位说明，不得生成假证书。
```

图片去重规则：

```text
1. `image-manifest.json` 必须增加 `visualType` 字段，例如 hero / people / environment / detail / product / factory / texture。
2. 每张图必须写明和其他图片的差异点。
3. 如果 3 张以上图片都是同一会议室、同一人物组合、同一角度，验收失败。
4. 如果图片只是裁切不同但主体相同，不能算不同图片。
```

### 动效交互规则

企业官网必须加入克制的滚动进入动效，让页面有活力，但不能做成花哨动画站。

必须实现：

```text
1. 内容块滚动进入视口时淡入上移：opacity 0 -> 1，translateY(24px-40px) -> 0。
2. 多个卡片或图片列表使用 stagger 错峰出现，间隔 80ms-140ms。
3. 首屏文字、CTA、图片可以有轻微首屏加载动效。
4. 导航、按钮、图片卡片需要有 hover 状态。
5. 移动端保留轻微 fade-up，减少位移和复杂动画。
6. 必须支持 prefers-reduced-motion；用户关闭动画时不播放复杂动效。
7. 宽幅图片、场景图、案例图、产品图等主要图片位必须有鼠标 hover 浮动质感。
```

推荐实现方式：

```text
1. 使用 IntersectionObserver 给 `.reveal` 元素添加 `.is-visible`。
2. CSS 中定义 transition，不依赖大型动画库。
3. main.js 负责移动端导航、滚动 reveal、返回顶部或轮播控制。
4. 如果使用轮播，必须可暂停、可手动切换，不能影响阅读。
```

图片 hover 浮动效果规则：

```text
1. 主要图片容器必须 `overflow: hidden`，图片使用 `object-fit: cover`，避免 hover 时溢出或变形。
2. 桌面端鼠标移入图片卡片、宽幅图、案例图、场景图时，图片内部应轻微放大，例如 scale(1.01) -> scale(1.035-1.06)。
3. 图片容器可以轻微上浮 translateY(-3px 到 -8px)、增强阴影或让底部渐变遮罩/标题浮层更清晰，但不得导致页面布局跳动。
4. 图片标题浮层应使用底部渐变遮罩承托，文字保持可读；不要把整张图片盖成厚重黑块。
5. 触屏端不能依赖 hover 才显示关键信息；移动端默认显示必要标题和说明。
6. `prefers-reduced-motion: reduce` 时必须取消图片缩放、上浮和复杂过渡。
7. 推荐参考实现：`.image-card img { transform: scale(1.01); transition: transform 500ms-700ms ease; } .image-card:hover img { transform: scale(1.045); }`。
```

滚动 reveal 判定标准：

```text
1. `.reveal` 元素在进入视口前必须是 opacity: 0，并带有 translateY(24px-40px)。
2. IntersectionObserver 触发后才添加 `.is-visible`，变为 opacity: 1 和 translateY(0)。
3. 首屏内元素可以在页面加载后立即触发淡入，但屏幕下方元素不能默认可见。
4. 禁止“默认可见，触发时再轻微动一下”的假 reveal；这会导致内容已经出现在屏幕里还在滚动。
5. 禁止为了避免截图空白而把 reveal 默认设为可见。正确做法是等待 JS 触发、给首屏元素单独处理，或在验收截图前等待动画完成。
6. 没有进入视口的下方内容，在初始页面检查时应保持隐藏状态。
```

禁止：

```text
1. 滚动劫持。
2. 大面积文字飞入、旋转、抖动。
3. 因动画导致首屏空白或内容不可读。
4. 所有元素同时出现，没有层次。
5. 默认可见再播放位移动画，冒充滚动进入效果。
```

## 11. 验收标准

生成完成后，AI 执行者必须检查：

- 网站目录是否存在。
- 所有 HTML 页面是否存在。
- 导航链接是否能跳转到对应页面。
- CSS 是否被引用。
- JS 是否被引用。
- 图片是否存在。
- 图片数量是否达到所选模板最低要求。
- 图片文件名是否符合所选模板图片清单。
- robots.txt、sitemap.xml、llms.txt、schema.jsonld 是否存在。
- 每个页面是否包含 title、description、keywords、语义化 H1/H2 和图片 alt。
- 页面中是否包含公司名称。
- 页面中是否包含主营业务。
- 页面中是否包含真实联系方式，或用户明确允许的占位联系方式。
- 首页是否有首屏、服务/产品、公司介绍、案例/应用场景、联系入口。
- 页面栏目是否符合所选模板。
- 首页板块是否符合所选模板。
- 页面是否横向撑满：首屏、图片区、分区背景、底部是否 full-width，而不是窄容器居中。
- 视觉风格是否符合所选模板，而不是复用旧站点样式。
- 是否输出设计执行卡，且最终视觉是否符合设计执行卡。
- 是否从固定色板库或品牌来源中锁定一套配色，并在设计执行卡中写明色板名称、颜色值和选择原因。
- 主题色是否真实落位到导航激活态、H1 关键词、CTA、卡片 icon/序号、section 标题、图片遮罩/轮播点、页脚重点链接等关键位置。
- 是否避免无理由默认蓝色；如果选择蓝色，是否说明蓝色比其他行业色板更适合当前企业。
- 是否实现滚动 reveal 动效、hover 状态和移动端可用交互。
- 是否实现主要图片位的 hover 浮动质感，且没有布局跳动、文字遮挡或移动端信息缺失。
- 移动端布局是否基本可用。
- 必须检查桌面端和移动端视觉效果，建议至少检查 1440px 桌面、768px 平板、390px 手机。
- 截图检查时必须确认：没有横向滚动、导航可用、图片不变形、首屏不空、移动端菜单正常、页面不是纯白卡片堆叠。
- 截图必须保存到 `_verification/`，不能只口头说检查过。
- 图片必须有 `assets/images/image-manifest.json`，不能只看图片数量。
- 模板符合度必须有 `_workflow/template-conformance.md`，不能只口头说“参考了模板”。
- 必须人工/视觉检查首页截图：如果首屏不如普通真实企业官网可信，不能因为文件齐全就交付。
- 证据文件内容必须和实际页面一致；例如报告写“真实业务场景图”，但截图里是扁平插画，则验收失败。
- 必须检查图片是否重复：如果多张图片主体、人物、会议室或构图高度相似，视为图片策略失败。
- 必须检查图片类型是否丰富：不能全是真人会议图，也不能全是抽象图或设备特写。

建议用脚本检查：

```text
页面文件数量
图片引用路径
导航 href
核心文案
CSS/JS 引用
图片数量
模板页面栏目
SEO/AIO 文件
移动端横向滚动
主题色落位
色板一致性
```

验收失败条件：

```text
1. 缺少联系方式却没有追问。
2. 未输出生成前确认摘要。
3. 没有明确所选模板。
4. 图片数量低于模板最低要求。
5. 页面栏目与所选模板不一致。
6. 服务行业、制造行业、企业工厂模板混用。
7. 生成结果明显复用旧网站结构或旧视觉。
8. 未生成 robots.txt、sitemap.xml、llms.txt、schema.jsonld。
9. 未做桌面端和移动端截图检查。
10. 未检查执行环境能力，或未输出设计执行卡。
11. 未生成 `_workflow/` 和 `_verification/` 证据目录。
12. 图片都是抽象 UI 插画，缺少行业现场感或公司业务语义。
13. 没有 `image-manifest.json`，无法说明图片来源、用途和提示词。
14. 没有 `template-conformance.md`，无法证明和所选模板的关系。
15. 页面主体被限制在中间窄容器，缺少 full-width 首屏、背景、图片或轮播。
16. 没有滚动 reveal 动效，页面静态僵硬。
17. 图片高度重复，或全站几乎都是同一类真人会议图片。
18. 主要图片卡片、宽幅图或案例图没有 hover 浮动质感，或 hover 时图片溢出、布局跳动、文字被遮挡。
19. 未锁定色板，或临时随机配色。
20. 主题色只出现在一个按钮上，没有进入导航、标题、卡片、图片区和页脚等关键位置。
21. 没有品牌色或参考色依据时，无理由继续默认蓝色。
22. 主色、辅助色、强调色混用，导致每个 section 像不同网站。
```

## 12. 固定启动提示词

以后可以直接对任意 AI 编程工具使用以下提示词：

```text
请按照《企业官网静态 HTML 生成工作流》执行，给下面这家公司生成一套可部署的企业官网。

你需要：
1. 先读取工作流文件，并输出【工作流读取确认】。
2. 检查当前工具是否具备文件读写、前端实现、视觉设计、图片生成/处理、浏览器预览/截图能力，并输出【执行环境检查】。
3. 如果缺少图片生成、浏览器预览或文件写入能力，必须明确说明限制和替代方案；不能假装已经完成。
4. 对公司资料、参考模板链接、参考网站、用户图片做【参考源检查】；没有打开过的链接，不得声称已经看过。
5. 把本次任务压缩成【本次强制执行清单】，后续按清单逐项执行和验收。
6. 判断是否缺少公司名称、联系方式、主营业务、目标客户、城市/服务区域等企业官网必要信息。
7. 如果缺少公司名称、主营业务、有效联系方式、城市或服务区域，必须先追问。
8. 联系方式不得默认占位；至少需要电话、微信、邮箱或其他有效触达方式中的一种，除非我明确回复允许占位。
9. 根据公司简介判断企业属于服务行业、制造行业、企业工厂/工业企业。
10. 根据类型选择固定模板库中的视觉参考模板；制造行业和企业工厂都适用时，按“卖产品/设备/加工”或“展示厂区/产能/项目交付”的核心目标判断。
11. 模板只作为视觉和结构参考，不复制源码。
12. 生成前必须输出【设计执行卡】，包含设计依据、Design Read、首屏构图、视觉关键词、配色、字体、图片策略、动效策略、禁止事项和移动端策略。
13. 配色必须按工作流的“受控配色与主题色落位”执行：优先使用品牌色/参考色，否则从行业固定色板库选择一套；不得无理由默认蓝色。
14. 设计执行卡必须写明色板名称、主色、辅助色、强调色、背景色、文字色、禁用颜色和主题色落位。
15. 生成页面时必须让主题色进入导航激活态、H1 关键词、CTA、服务/产品 icon、卡片 hover、图片遮罩/轮播点、统计条或页脚重点链接。
16. 生成前必须输出确认摘要：工作流读取确认、执行环境检查、参考源检查、执行清单、设计执行卡、企业类型、所选模板、色板名称、主题色落位、页面栏目、首页板块、图片清单、缺失信息。
17. 自动生成中文页面文案。
18. 页面必须横向撑满浏览器宽度：首屏、背景色块、宽幅图片、轮播、底部区域都要 full-width；文字可以使用舒适内层容器，但不能整站像被塞在中间窄框里。
19. 自动生成相关网站图片，图片数量必须达到所选模板最低要求；如果我提供真实图片，必须优先使用真实图片。
20. 图片必须有类型配比和去重：不能全是真人会议图，必须混合首屏大图、服务/产品/工厂场景、空间/环境/景色、细节小配图或辅助纹理；重复构图视为失败。
21. 必须加入克制滚动动效：内容滚动进入时淡入上移，卡片错峰出现，按钮和图片有 hover 状态，并支持 prefers-reduced-motion。主要图片位必须保留鼠标 hover 浮动质感：图片轻微放大，卡片可轻微上浮或增强阴影，标题浮层清晰，不能布局跳动。
22. 输出纯静态 HTML 网站，不要后端，不要数据库。
23. 网站必须包含页面跳转。
24. 最终输出到 outputs/company-site。
25. 必须生成 robots.txt、sitemap.xml、llms.txt、schema.jsonld，方便搜索引擎和 AI 抓取理解。
26. 生成后必须输出【完成后验收】，检查页面、链接、图片、样式、图片数量、模板符合度、设计执行卡符合度、配色一致性、主题色落位、SEO/AIO 文件和桌面/移动端截图是否完整。
27. 交付目录必须包含 `_workflow/`、`_verification/` 和 `assets/images/image-manifest.json`；缺少这些证据文件时，不能声称完成。
```

## 12.1 简化使用方式

当用户在新对话中只说“做一个企业官网”“根据下面公司简介生成官网”“我要给这家公司做官网”等同类请求，并粘贴公司简介时，AI 执行者必须自动进入本工作流，不得要求用户再次解释流程。

用户最简输入可以是：

```text
做一个企业官网，公司简介如下：
【公司简介】
```

AI 执行者收到后必须自动执行：

```text
1. 读取本工作流。
2. 检查当前执行环境能力，并说明缺失能力的处理方式。
3. 判断必要信息是否缺失，缺失则先追问。
4. 判断企业类型并选择模板。
5. 使用当前工具可用的设计能力或内置审美规则形成设计执行卡。
6. 生成多页面静态官网。
7. 输出完成后验收。
```
